Experiment: Integrated Arithmetic Benchmark for LLMs: Operand Size, Operation Type, Prompt Format, Tokenization, and Error-Mechanism Analysis

Original experiment design query
How do LLMs perform arithmetic — how does accuracy vary with operand size, operation type, and prompt format, and what do the error patterns reveal about how they compute?
Experiment Design Description
This experiment measures how LLM arithmetic performance changes with operand size, operation type, and prompt format using three complementary benchmark blocks: controlled synthetic symbolic arithmetic, multilingual arithmetic word problems (MGSM), and grouped arithmetic-expression problems (MATH401). The design uses fixed model versions, deterministic decoding, pre-registered parsers, exact canonical answer formats, and paired within-item prompt comparisons so that accuracy differences can be attributed cleanly to experimental factors. In addition to exact-match accuracy, the experiment records digit-level, length-level, relative-error, parse-failure, and taxonomy-based error signals, plus tokenizer-derived representation covariates. A limited open-model mechanistic sub-study connects observed arithmetic behavior to internal components using activation patching and answer probes, but all primary conclusions remain valid without white-box access.
Plain-Language Experiment Design Description
This study is designed to answer a focused question: when large language models do arithmetic, how does their accuracy change as the numbers get bigger, as the operation changes, and as the prompt is worded differently, and what do their mistakes suggest about the way they are computing. Before running anything, the researchers preregister the main hypotheses, freeze the model roster and prompt templates, and define exactly how answers will be parsed and scored, so they do not change the rules after seeing the results. The benchmark has three separate parts, each analyzed on its own rather than merged into one score: Block A is a tightly controlled synthetic arithmetic test, Block B is MGSM multilingual arithmetic word problems, and Block C is the MATH401 set of calculator-style expression problems. Block A is the core causal test: it contains exactly 12,000 unique items, balanced across addition, subtraction, multiplication, and division and across operand sizes from 1 to 6 digits before the decimal point, with each operand written as a signed decimal with exactly three digits after the decimal; answers are computed using exact decimal arithmetic, then rounded to exactly two decimal places with standard round-half-up rules. Each Block A item is asked in four matched prompt formats: a direct symbolic question, the same question with operator words such as "plus" or "divided by," a step-by-step chain-of-thought version that must end with a fixed answer line, and a version where a system message tells the model to output only a fixed-format number. Block B uses all 250 MGSM items in each of 11 languages, preserving the original wording exactly, and adds carefully annotated labels such as which operations are used, how many reasoning steps are needed, and how large the numbers get; each item is tested with direct answering, reasoning in the source language, reasoning in English, and, for non-English items, solving a fixed English translation. Block C uses all 401 MATH401 items with their published groups kept intact, including easy and hard groupings, and tests prompt variants like leaving the problem unchanged, adding "Calculate:", or adding a zero-shot step-by-step instruction, while scoring answers against high-precision computed targets normalized to four decimal places.

1.1

Primary research questions and preregistered hypotheses

The experiment addresses four linked questions: how arithmetic accuracy varies with operand size, how it differs by operation, how prompt format changes performance, and what error patterns imply about computation. Preregistering these questions and predictions is essential because arithmetic evaluations are highly sensitive to benchmark composition, parser choice, and prompt wording.
Implementation details
Create a preregistration package before any model queries. Save it as both a human-readable Markdown file and a machine-readable JSON file. Include these exact primary hypotheses. H1: on controlled symbolic arithmetic, exact-match accuracy decreases as operand size increases. H2: at matched size, multiplication and division are harder than addition and subtraction. H3: prompt-format effects depend on task family: direct-answer prompting will be strongest or comparable on one-step symbolic arithmetic, while explicit reasoning prompts may help more on multilingual word problems and may help or hurt on expression-calculation tasks. H4: as operands grow, error distributions shift from isolated same-length digit substitutions toward wrong-length outputs, decimal-placement errors, token-boundary or misalignment errors, and catastrophic magnitude failures. H5: tokenizer fragmentation and numeral representation partially mediate length effects. H6: in open-model mechanistic sub-studies, if arithmetic is implemented by sparse heuristics rather than a unified robust algorithm, a relatively small subset of components should explain a large fraction of correct-answer logit mass. Define primary endpoints as follows: Block A exact-match to canonical two-decimal output, Block B exact-match to gold Arabic numeral, Block C correctness under absolute error less than 1e-3 after four-decimal normalization. Define secondary endpoints: digit-match, dlength, non-number rate, absolute error, relative error, per-category error frequencies, tokenization covariates, and paired prompt differences. State explicitly that mechanistic interpretation is triangulated from multiple sources and that behavior alone is insufficient to prove a unique internal algorithm.

3.1

Operand representation and fixed interpretation

Because prior descriptions of this benchmark style can be ambiguous, the experiment must commit to one exact operand format and never deviate. Fixed-width finite decimals allow exact Decimal arithmetic, consistent prompting, and deterministic scoring.
Implementation details
Define k as the number of digits in the integer part. Use k in {1,2,3,4,5,6}. Generate operands with exactly k integer digits and exactly three fractional digits. For k=1, sample the integer part uniformly from 1 through 9 inclusive. For k>1, sample the integer part uniformly from 10^(k-1) through 10^k - 1 inclusive. Sample the fractional part uniformly from 0 through 999 inclusive and format it with exactly three digits using leading zeros. Independently assign each operand a sign with probability 0.5 positive and 0.5 negative. For division, reject and resample any denominator whose full signed decimal value is exactly 0.000. Render each operand as optional minus sign plus integer part plus '.' plus three fractional digits. Use a fixed RNG seed 20250201. Record RNG type and library version in metadata.

Infrastructure failure policy

Infrastructure failures should be retried systematically, but semantic failures such as malformed outputs should never be retried because they are part of model behavior.
Implementation details
